# Contributing to MIC

Thank you for considering contributing.

MIC is experimental, so clear examples, criticism, and security feedback are especially valuable.

## Good Contributions

Useful contributions include:

- clearer examples
- better documentation
- validator improvements
- MIC Core safety checks
- test contracts
- threat model improvements
- audit record improvements
- terminology cleanup

## Design Rules

Please keep these boundaries intact:

- MIC Core verifies and supervises.
- Engines produce evidence.
- Application workers apply business state.
- Engines should not directly mutate application databases.
- Unknown tasks and actions should fail closed.

## Security Issues

Please do not publicly disclose serious security issues before maintainers have had time to review them.

Report security issues to:

support@esluka.co.uk
